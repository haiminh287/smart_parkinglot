## 1. TL;DR — Đọc trong 60 giây

> 1. Vehicle spawns at **(47, 0.1, 0)** facing **-X** (yaw -90°), drives straight 8.5m to GATE-IN-01 waypoint at **(38.5, 0.1, 0)** — one-node path, no turns.
> 2. After QR check-in, `ProceedFromGate()` uses BFS pathfinding from gate waypoint → lane node (32, 0.1, 0) → inner lane nodes → slot entrance. The vehicle **crosses through the gate structure and the east curb** (no physics, uses MoveTowards).
> 3. **BarrierController.barrierArm is a SerializeField** — the gate arm is procedurally generated by `CreateGate()` but there is **no code auto-linking** the generated arm to the BarrierController. Must be done in Inspector or there's a missing initialization step.

#### `FollowPath()` (lines 152–177) — The movement engine

```csharp
Vector3 targetPos = targetNode.transform.position;
Vector3 direction = targetPos - transform.position;
direction.y = 0f;  // flatten to horizontal

Quaternion targetRot = Quaternion.LookRotation(direction);
transform.rotation = Quaternion.Slerp(transform.rotation, targetRot, Time.deltaTime * rotationSpeed);
transform.position = Vector3.MoveTowards(transform.position, targetPos, moveSpeed * Time.deltaTime);

if (dist < 0.3f) currentPathIndex++;  // arrival threshold = 0.3m
```

- **No physics** — uses `MoveTowards` (passes through all colliders/geometry).
- Rotation via `Slerp` with `rotationSpeed = 3f` — smooth but can be delayed.
- Speed: `moveSpeed = 5f` (5 m/s = 18 km/h).
- Arrival threshold: **0.3m**.

#### `ParkCoroutine()` (lines 224–261)

## 4. Complete Entry Flow Path

### Phase 1: Spawn → Gate (single straight line)

```
Spawn (47.0, 0.1, 0)  ────────→  GATE-IN-01 waypoint (38.5, 0.1, 0)
        facing -X                    8.5m straight along -X
        on entry road               at stopping bay center
```

**Duration:** 8.5m ÷ 5 m/s = **~1.7 seconds**

### Phase 2: Wait at gate

- State: `WaitingAtGate`
- `HandleVehicleAtEntry()` adds to `vehiclesWaitingAtGate` list
- Waits for `CheckInWaitingVehicle(plate)` call (triggered by ESP32 QR scan)

### Phase 3: Gate opens → Drive to slot

```
GATE-IN-01 (38.5, 0.1, 0)
  │
  ↓ connected to
Lane[7] (32.0, 0.1, 0)
  │
  ↓ lane chain
Lane[6] (22.86, 0.1, 0) → Lane[5] (13.71) → ... → Lane[N] nearest to slot
  │
  ↓ connected to
SlotEntrance (slot.x, 0.22, slot.z ± 3.5)   ← offset from slot center
  │
  ↓ ParkCoroutine()
Slot center (slot.x, slot.y, slot.z)
```

### Phase 4: Parking

- 0.5s rotation to align with slot
- 2.0s linear interpolation into slot center
- State → `Parked`

## 5. ⚠️ Gotchas & Known Issues

- [x] **[NOTE]** Path from gate (38.5) to lane[7] (32.0) crosses through **gate structure at x=35** and **east curb at x=35**. No collision — `MoveTowards` ignores physics. Visual clipping will occur.

- [x] **[NOTE]** **BarrierController.barrierArm** is a `[SerializeField] Transform` — the "GateArm" cube is procedurally generated by `CreateGate()` but **no code auto-assigns it** to the BarrierController. If not manually wired in Inspector, `barrierArm == null` and barrier animations are silently skipped (the `Update()` early-returns on null).

- [x] **[NOTE]** Spawn → gate path has **only 1 waypoint** (the gate node itself). No intermediate waypoints on the entry road. Works fine because it's a straight line, but means there's no queueing logic on the approach road.

- [x] **[NOTE]** `RecognizePlateAtGate()` starts **simultaneously** with `ProceedFromGate()`. It waits only 2 frames. The vehicle is already moving by then — the plate may have shifted by the time the snapshot is taken.

- [x] **[NOTE]** ANPR virtual camera at (43.5, 1.0, 0) is 5m behind the gate waypoint (38.5). When vehicle is at (38.5, 0.1, 0) facing -X, its rear plate is at ~(40.8, 0.48, 0). Camera sees it at 2.7m distance. Once `ProceedFromGate()` starts, the car moves -X at 5m/s — the plate moves out of frame within ~1 second.

- [x] **[NOTE]** `vehicleSpawnPoint` is a serialized reference. If null, fallback is (47, 0.1, 0). Cannot determine actual Inspector value from code alone.
